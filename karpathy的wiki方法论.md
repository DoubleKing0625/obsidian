# Karpathy 的 Wiki 方法论

> 来源：`LLM Wiki: A pattern for building personal knowledge bases using LLMs`

## 核心想法

传统的 RAG 或文件上传式问答，是在每次提问时临时从原始资料里检索片段，再拼出答案。问题是：知识不会沉淀。每次遇到复杂问题，LLM 都要重新发现、重新关联、重新综合。

这个方法论的核心是：不要只让 LLM 在查询时读取资料，而是让 LLM 持续维护一个持久化的个人 wiki。

也就是说，当你加入一篇文章、一本书的一章、一份报告或一段访谈时，LLM 不只是把它放进索引，而是主动把其中的关键信息整理进现有知识库：

- 创建或更新主题页
- 创建或更新人物、概念、项目等实体页
- 加强已有总结
- 标出新资料和旧说法之间的矛盾
- 补充交叉链接
- 把一次问答中形成的洞见沉淀成新页面

最终，wiki 不是聊天记录的副产品，而是一个会随着资料和问题不断增厚的知识制品。

可以把关系理解成：

- Obsidian 是 IDE
- LLM 是程序员
- wiki 是代码库
- 原始资料是不可随意改写的 source of truth

## 三层架构

### 1. 原始资料层

原始资料是你收集的文章、书摘、论文、图片、网页、会议记录、访谈稿等。

原则：

- 原始资料尽量保持不可变
- LLM 可以读取、引用、总结它们
- 不要让 LLM 随意改写原始资料
- 原始资料承担“证据”和“出处”的角色

### 2. Wiki 层

Wiki 是由 LLM 维护的一组 Markdown 页面。

它可以包括：

- 主题页
- 概念页
- 人物页
- 项目页
- 资料摘要页
- 对比分析页
- 研究综述页
- 当前判断或阶段性结论

这一层是 LLM 的主要工作区。LLM 负责创建页面、更新页面、维护链接、合并重复概念、标出冲突和空白。

### 3. Schema 层

Schema 是给 LLM 的维护说明书。对 Codex 来说，可以是 `AGENTS.md`；对 Claude Code 来说，常见是 `CLAUDE.md`。

它应该说明：

- wiki 的目录结构
- 命名规则
- 页面模板
- frontmatter 约定
- 引用和出处规则
- 新资料如何 ingest
- 查询结果何时写回 wiki
- 何时更新 index 和 log
- 哪些文件只读，哪些文件由 LLM 维护

这个 schema 会随着你的使用习惯一起演化。它的作用是把 LLM 从“泛泛聊天助手”约束成“有纪律的 wiki 维护者”。

## 三个核心操作

### Ingest：摄入新资料

当你加入一份新资料时，LLM 应该完成一轮整理。

推荐流程：

1. 阅读原始资料
2. 提取核心观点、事实、人物、概念和问题
3. 写一篇资料摘要页
4. 更新相关主题页、概念页、人物页
5. 补充双链和反向链接
6. 标记和旧内容的矛盾或修正
7. 更新 `index.md`
8. 在 `log.md` 追加记录

一份资料可能影响 10 到 15 个页面。关键不是“多写一篇摘要”，而是把新知识接入既有网络。

### Query：查询和综合

提问时，LLM 不应该只从原始资料里临时检索，而应该优先读取 wiki：

1. 先读 `index.md` 找相关页面
2. 再读相关主题页、概念页、资料摘要页
3. 必要时回到原始资料核对
4. 给出带出处的综合回答
5. 如果回答形成了有价值的新洞见，就写回 wiki

一个好问题本身也可以成为知识库的增长点。

例如：

- “这些文章对长期主义的分歧在哪里？”
- “某个作者的观点在最近三篇文章里有没有变化？”
- “这几个产品的定位差异是什么？”
- “我最近的日记反复出现了哪些心理模式？”

这些回答不应该消失在聊天窗口里，而应该变成可链接、可复用、可继续演化的页面。

### Lint：健康检查

Wiki 会增长，也会变乱。所以需要周期性检查。

LLM 可以定期查找：

- 页面之间的矛盾
- 已经过时的判断
- 没有入链的孤立页面
- 被频繁提到但还没有独立页面的概念
- 缺少出处的断言
- 重复页面或重复概念
- 需要进一步搜索或阅读的资料空白

这相当于给知识库做重构。

## 两个关键文件

### index.md

`index.md` 是内容导向的目录。

它应该列出 wiki 中的重要页面，并给出一句话说明。可以按类别组织：

- 主题
- 概念
- 人物
- 项目
- 资料摘要
- 研究问题
- 阶段性结论

LLM 在回答问题前先读 index，再决定深入哪些页面。中等规模的知识库里，这比一上来就做复杂向量检索更简单、透明、可维护。

### log.md

`log.md` 是时间导向的操作记录。

建议用统一格式追加，例如：

```md
## [2026-06-24] ingest | LLM Wiki
```

它记录：

- 新资料摄入
- 重要查询
- lint 检查
- 大规模重构
- 关键判断变化

好处是 LLM 可以快速知道最近做过什么，人也可以回看知识库的演化过程。

## 为什么这种方法有效

维护知识库最消耗人的部分，不是阅读和思考，而是整理：

- 补链接
- 更新摘要
- 维护目录
- 合并重复信息
- 标记冲突
- 把新旧知识重新综合

人很容易因为这些琐碎工作放弃 wiki。LLM 的价值正好在这里：它不会厌倦这些维护工作，也可以一次性触碰很多文件。

人的工作应该是：

- 选择资料
- 提出问题
- 判断什么重要
- 校正方向
- 做最终理解

LLM 的工作应该是：

- 归档
- 总结
- 链接
- 对比
- 维护一致性
- 把探索沉淀成页面

## 适合的使用场景

这个方法适合任何需要长期积累、反复综合的领域：

- 个人成长：目标、健康、心理模式、日记复盘
- 研究项目：论文、报告、文章、观点演化
- 读书：章节、人物、主题、情节线、概念网络
- 商业分析：竞品、客户访谈、会议记录、项目材料
- 课程学习：讲义、作业、概念、错题和总结
- 旅行规划：地点、路线、预算、体验记录
- 兴趣研究：任何会持续深入的爱好或问题域

## 和 Memex 的关系

这个想法和 Vannevar Bush 在 1945 年提出的 Memex 有相似精神：一个私人的、主动维护的、带有联想路径的知识库。

区别在于，过去最大的问题是“谁来维护这些连接”。现在 LLM 可以承担大部分维护工作。

## 对这个 vault 的启发

如果要在当前 Obsidian vault 里实践，可以从一个简单版本开始：

```text
raw/          原始资料，只读
wiki/         LLM 维护的主题页、概念页、摘要页
index.md      全局索引
log.md        操作日志
AGENTS.md     给 Codex 的维护规则
```

初始规则可以很简单：

- 新资料先放入 `raw/`
- 每次摄入都至少生成一篇摘要页
- 重要概念用 `[[双链]]` 连接
- 有价值的查询结果写回 `wiki/`
- 每次操作更新 `index.md` 和 `log.md`
- 定期要求 LLM 做一次 lint

重点不是一开始设计完美系统，而是让 wiki 开始复利。
